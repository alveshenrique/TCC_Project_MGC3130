When to use the GestIC SDK

The GestIC SDKs API provides an synchronous highlevel communication with the MGC3130.  It is portable to different platforms and highly customizable.

Limitations of the GestIC SDK

The SDK currently doesn’t directly support writing applications that require asynchronous communication with immediately returning functions.

However the development of such applications could still benfit from the SDK as a reference on how to interact with an GestIC device.

Static API vs Dynamic API

The SDK provide two slightly different APIs to applications for different use cases: The Dynamic API for use mainly with DLLs and the Static API for use mainly in embedded development.

Here is an overview of which API could be used when:

Dynamic APIStatic API
Dynamic Linking ¹YesNo
Static LinkingYes ²Yes
EmbeddedNot recommended ²Yes

¹ Dynamic Linking is enabled by setting the define GESTIC_API_DYNAMIC on the compiler command line.  It is required both when compiling the library and when using the library.  While building the API library a second define GESTIC_API_EXPORT has to be set.

² The dynamic API is not provided on those configurations per default it requires the define GESTIC_HAS_DYNAMIC.  GESTIC_HAS_DYNAMIC unlike GESTIC_API_DYNAMIC does not block access to the static API.

Dynamic API

The dynamic API is designed for use in a dynamically linked GestIC library (gestic.dll on Windows or gestic.so on Linux).  As those file are easy to replace with different versions it hides implementation details behind function calls.  The API is designed to be binary compatible to earlier versions which means that using a newer GestIC library is save to use with an application that was developed for an older version of the API.

Note

The backward comptibility is only effective on binaries.  There are a few enumerations that have no longer an effect, that were removed from the API and are ignored in the implementation.

Static API

The static API is designed for use with embedded applications.  Embeded platforms are short on resources and often applications don’t use dynamic memory management at all.  They also provide different possibilities for communication with external hardware.

The static API respects this in avoiding the the overhead of the dynamic API and exposing the internal implementation details to some degree.  It is also highly configurable and allows custom implementations for the IO-communication.

Differences between static and dynamic API

In the dynamic API gestic_t is an opaque type that could only used with special functions that are not part of the static API.  Those functions are gestic_create and gestic_free for allocation and freeing of the gestic_t instance and the functions that provide access to the fields of the data output buffer.

Beside those and the deprecated functions that are skipped in the static API both APIs provide the same functions.
